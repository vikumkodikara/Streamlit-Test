1. Introduction to Streamlit
Streamlit  is  an  open-source  Python  framework  that  lets  data  scientists,  AI/ML  engineers,  and 
researchers turn Python scripts into interactive web applications in minutes — with no front-end or  
web development knowledge required. It follows a pure-Python, script-first philosophy: you write top-
to-bottom Python, and Streamlit renders it as a live, reactive web app automatically.
For Final Year Projects (FYP), Streamlit is the ideal Proof-of-Concept (PoC) tool because it bridges  
the gap between a trained machine learning model, a data analysis pipeline, or an AI system and a  
demonstrable,  interactive  interface  that  supervisors,  examiners,  and  industry  partners  can  
immediately use and evaluate.
1.1 Why Streamlit for Your FYP PoC?
Criterion Traditional Web App Streamlit PoC
Development Time Weeks (HTML/CSS/JS + backend) Hours (pure Python)
Required Skills Full-stack knowledge Python only
ML/AI Integration Manual API wiring Direct function calls
Iteration Speed Slow (rebuild/redeploy) Instant (hot-reload)
Deployment Complex DevOps setup One command / free hosting
Examiner Demo Requires server setup Share a URL
1.2 How Streamlit Works — The Execution Model
Understanding Streamlit's execution model is essential. Every time a user interacts with a widget  
(clicks a button, moves a slider), Streamlit re-runs the entire script from top to bottom. This is  
intentional and keeps the app reactive.
# Every interaction re-runs this entire file
import streamlit as st
 
st.title('My FYP Demo')          # Rendered on every run
name = st.text_input('Your name')  # Widget — triggers re-run on change
st.write(f'Hello, {name}!')       # Shows updated output immediately
Key Concept: Re-run Model
Streamlit is NOT event-driven like React or Flask. The full script runs on EVERY interaction.
Use st.cache_data / st.cache_resource to avoid recomputing expensive operations on every 
run.
Use st.session_state to preserve values across re-runs (covered in Session 7).

    page_title='FYP Demo',
    page_icon=':brain:',
    layout='wide',          # 'centered' or 'wide'
    initial_sidebar_state='expanded'
)

# Run: streamlit run app.py
# Opens at http://localhost:8501
Important: st.set_page_config() Rules
Must be the FIRST Streamlit command in your script — before any other st.* call.
Can only be called ONCE per app (not on individual pages in a multi-page app).
layout='wide' is recommended for data-heavy FYP dashboards.

4. Input Widgets
Widgets are the interactive controls that collect user input in a Streamlit app. Every widget returns a  
Python value that you can immediately use in your code. When a user interacts with a widget, the  
script re-runs and the new value is available.
4.1 Text and Number Inputs
import streamlit as st
 
# Text input
name = st.text_input('Enter your name', value='Student', max_chars=50)
 
# Password (hidden input)
api_key = st.text_input('API Key', type='password')
 
# Multi-line text area
notes = st.text_area('Research Notes', height=150,
                     placeholder='Describe your methodology...')
 
# Number input (with step and bounds)
threshold = st.number_input('Classification Threshold',
                            min_value=0.0, max_value=1.0,
                            value=0.5, step=0.01,
                            format='%.2f')
 
st.write(f'Classifying as positive if probability > {threshold}')

7. Session State and Reactivity
Session state is one of the most important — and frequently misunderstood — features in Streamlit.  
Because the entire script re-runs on every widget interaction, you need a mechanism to persist values 
across those re-runs. st.session_state provides this per-user, in-memory persistence.
7.1 Understanding the Problem
# PROBLEM: Without session_state, count resets to 0 on every re-run
import streamlit as st
 
count = 0   # This is reset to 0 EVERY TIME the script runs
if st.button('Increment'):
    count += 1   # This never actually persists
st.write(f'Count: {count}')   # Always shows 0
# SOLUTION: Use session_state
import streamlit as st
 
# Initialise (only runs if key doesn't exist)
if 'count' not in st.session_state:
    st.session_state.count = 0
 
if st.button('Increment'):
    st.session_state.count += 1
 
st.write(f'Count: {st.session_state.count}')  # Persists correctly

9. Caching and Performance Optimisation
Since Streamlit re-runs the entire script on every interaction, expensive operations (loading datasets, 
training models, calling APIs) would repeat unnecessarily. Caching is the primary performance tool in 
Streamlit.
9.1 @st.cache_data — For Serialisable Data
Use cache_data for functions that return DataFrames, NumPy arrays, lists, dicts, or any serialisable  
Python object:
import streamlit as st
import pandas as pd
import time
 
@st.cache_data   # Caches the return value
def load_dataset(filepath: str) -> pd.DataFrame:
    '''Loads CSV — cached so file is read only once'''
    time.sleep(2)   # simulate slow IO
    df = pd.read_csv(filepath)
    return df
 
@st.cache_data(ttl=3600)   # Cache expires after 1 hour
def fetch_live_data(api_url: str) -> dict:
    import requests
    return requests.get(api_url).json()
 
@st.cache_data(max_entries=5)  # Limit cache size
def preprocess(df: pd.DataFrame, scaler_type: str) -> pd.DataFrame:
    # heavy preprocessing
    return df
 
# Usage — only runs once; subsequent calls return cached result
df = load_dataset('data/medical_records.csv')
st.dataframe(df.head())
9.2 @st.cache_resource — For Non-Serialisable Resources
Use cache_resource for ML models, database connections, and other objects that cannot (or should 
not) be serialised:
import streamlit as st
import pickle

def load_model(path: str):
    '''Load ML model once — shared across all users'''

        return pickle.load(f)
 
@st.cache_resource
def get_db_connection():

    return sqlite3.connect('fyp_database.db', check_same_thread=False)

Performance Best Practices
Always cache dataset loading — even a 1 MB CSV takes 50-200ms to parse on every run.
Always cache model loading — a 100MB model file takes 2-10 seconds to load.
Use ttl= for data that changes over time (API responses, database queries).
Avoid caching functions with side effects (e.g., writing to files inside a cached function).
Use st.cache_resource for Transformers / TensorFlow / PyTorch models — they are not

12.4 FYP Demonstration Checklist
Before Presenting Your FYP Demo
1. Test with fresh data — your examiner WILL upload their own file.
2. Add loading spinners on all slow operations (model inference, DB queries).
3. Show metric cards prominently — accuracy, precision, recall, F1.
4. Include an 'About' section explaining your model and dataset.
5. Test on mobile — examiners may view on phones; use layout='wide' for desktop.
6. Add meaningful error messages, not just raw Python tracebacks.
7. Deploy to Streamlit Community Cloud before demo day — do not demo on localhost.
8. Keep a backup of your deployed URL and your model file in Google Drive.
9. Test the deployed version — it may behave differently from your local machine.
10. Clear session state before demos to start with a clean slate.
